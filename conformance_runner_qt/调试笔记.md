## 未解决的问题

### sse-retry — QtHttpSseTransport（QNetworkAccessManager 版）

- **问题**：`QtHttpSseTransport` 在工作线程上使用 `QNetworkAccessManager` 做 SSE GET 时，服务端关闭 TCP 连接后，`QNetworkReply` 的以下信号均不触发：
  - `finished`
  - `readChannelFinished`
  - `errorOccurred`
- **根因**：`QNetworkAccessManager` 在工作线程（`QThread`）上的内部 I/O 循环无法可靠检测 SSE 无限流连接的 TCP 断开。这是 Qt 网络栈的行为限制——主线程上相同代码可以工作，但迁到工作线程后信号丢失。
- **涉及文件**：`mcp_qt_transport/src/QtHttpSseWorker.cpp`、`mcp_qt_transport/src/QtHttpSseWorker.h`
- **已尝试的方案**：
  1. 健康检查定时器（50ms 间隔检测 `isFinished`/`error`）→ 信号不触发，检测失效
  2. `readChannelFinished` 信号连接 → 同样不触发
  3. 数据到达时间戳 + 空闲超时 → `handleSseReadyRead` 在 POST 响应体到达时触发，但 SSE GET 自己的 `readyRead` 未必触发
  4. 预留 `m_sseReply` 不替换 + 等待原始 reply 信号 → 信号本身就不触发
- **对比**：`HttpSseTransport`（cpp-httplib/CURL）用后台线程阻塞读可以可靠检测，因为它直接调用 `read()` 等待数据，TCP 断开时 `read()` 返回 0 或错误。但 `QNetworkAccessManager` 封装了底层 socket，不暴露读操作给调用方。
- **决策**：接受此场景不通过，已加入 `conformance-baseline.yml` 预期失败列表。Qt 平台限制导致需要架构级重构（换用 httplib），收益不大。
- **修复方向**（如需修复）：
  - 让 `QtHttpSseTransport` 改用 cpp-httplib + CURL（非 Qt 版）替代 QNAM，保持在工作线程上
  - 或者留在主线程用 QNAM + 主事件循环（但会阻塞 UI）

### 7. 运行测试时路径问题

- **问题**：在 Windows 上使用相对路径 `./build/...` 运行客户端时，Node.js 的子进程找不到二进制文件
- **解决方法**：始终使用**绝对路径**：
  ```bash
  QT_BINARY="f:/B_My_Document/GitHub/mcp-cpp-agent/build/conformance_runner_qt/mcp_client_conformance_qt.exe"
  ```
- **根因**：npx 的 conformance 框架在 Windows 上用 `shell: true` 启动子进程，路径解析行为与直接执行不同

### 8. 构建时的链接问题

- **问题**：Qt 版本 conformance 客户端和非 Qt 版本不能同时链接 `ScenarioRegistry.cpp`
- **原因**：`ScenarioRegistry.cpp` 中的函数声明由不同源文件实现（非 Qt 版用 `ScenarioHandlersBasic.cpp` + `ScenarioHandlersAuth.cpp`，Qt 版用 `ScenarioHandlersAuthQt.cpp` 自包含全部实现）
- **解决方法**：Qt 版本的 CMakeLists.txt 只包含 `ScenarioHandlersAuthQt.cpp`（不包含 `ScenarioHandlersBasic.cpp` 和 `ScenarioHandlersAuth.cpp`），且 `ScenarioHandlersAuthQt.cpp` 中自包含了全部 6 个处理函数的实现

## 架构说明

```
conformance_runner_qt/
├── CMakeLists.txt              # Qt 版本构建配置（链接 mcp_qt_client + mcp_core）
├── main_qt.cpp                 # 入口：解析环境变量 → 创建 RunnerConfig → 调用 runScenario
├── ScenarioHandlersAuthQt.cpp  # 所有场景处理函数（使用 McpQtClient API 或底层 HttpSseTransport）
├── 调试笔记.md                  # 本文件

conformance_runner/             # 非 Qt 版本（参考实现）
├── ScenarioHandlersBasic.cpp   # initialize/tools_call/sse-retry/elicitation（使用底层 session API）
├── ScenarioHandlersAuth.cpp    # auth 流程（使用底层 session API）
├── ScenarioHandlersList.cpp    # tools_list（使用 ConsoleStdioTransport）
├── ScenarioRegistry.cpp        # 场景名称 → 处理函数映射表
└── ScenarioRegistry.h          # runScenario / registeredScenarioNames 声明
```

测试客户端通过 `MCP_CONFORMANCE_SCENARIO` 环境变量获取场景名称，在 `runScenario()` 中查找注册表，调用对应处理函数。服务端 URL 由 conformance 框架通过命令行参数传递。
